Overview

Context for non-experts: what LI-COR does, and what problem I set out to solve.

Back to top

LI-COR Biosciences builds scientific instruments that measure gases like CO₂ and H₂O in environmental research. These instruments output large time-series files (TGA) that technicians review to verify performance and troubleshoot issues. Historically, this review required reformatting data in Excel and hand-building multiple charts—time-consuming, inconsistent, and hard to repeat.

My project replaces that manual process with a dedicated tool that ingests files, applies domain-specific logic (e.g., detecting warm-up vs. running), and renders clear visual summaries so support engineers can act quickly.

Role & Day-to-Day

How I worked across coding, UX, and collaboration.

Back to top
  • Develop: Python pipelines to parse, validate, and visualize TGA data with Matplotlib.
  • Design: A GUI that exposes the right controls (thresholds, filters, spans) without overwhelming users.
  • Collaborate: Weekly feedback loops with support engineers to test with real files and iterate quickly.
  • Validate: Edge-case handling (invalid spans, restarts, temperature thresholds) to keep charts trustworthy.

Project Highlights

A few focused wins that made the team’s workflow faster.

Back to top

Warm-up/Running Spans

Shaded bands with adjustable thresholds. Auto-detect startup (laser cavity/enclosure heating) and running states; prune invalid spans; let users tune rules without re-parsing.

One-Click Visuals

Purpose-built charts beat ad-hoc Excel. Generate consistent plots for common diagnostics and highlight only actionable trends.

Human-Centered Tweaks

Surface the controls the team actually uses (threshold sliders, span toggles), improve labeling, and add tooltips for clarity.

What I Learned & How It Transfers

Reflection tied to future contexts.

Back to top
  • Translate complexity: Turn raw sensor data into insights for non-experts—vital for any product role.
  • Iterate with users: Early, frequent feedback prevents over-engineering and accelerates delivery.
  • Own ambiguity: Break open-ended problems into testable pieces; document decisions for repeatability.
  • Quality mindset: Validate against edge cases so outputs are trusted in real support scenarios.

These habits transfer directly to future software roles—especially where domain expertise and UX meet (scientific tools, data platforms, instrumentation).

Advice for Future Interns

Concrete tips I wish I had on day one.

Back to top
  1. Start with the user. Shadow support workflows; collect pain points before designing features.
  2. Prototype fast. Ship a scrappy chart or toggle; validate direction before polishing.
  3. Name edge cases. Define warm-up/running logic early; write test files that trigger failures.
  4. Document decisions. A lightweight changelog saves the team (and future you) hours.
